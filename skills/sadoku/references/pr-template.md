# PR 説明文モードの本体

このファイルが PR 説明文モードの全実体。`sadoku` SKILL.md からこのファイルを読み込み、以下の手順で章ごとに提案する。

## 進め方 (全 8 ステップ)

1. issue / 計画から「課題」と「DoD」を抽出 (DoD は testable な粒度に分解)
2. diff (または計画) を読んで「実装の流れ」を numbered で構造化、各 step に File / Change / Review を埋める
3. scope 外発見を「実装中に分かったこと」に切り出し、別 issue 化を提案
4. 「検証」コマンドと手動シナリオを列挙
5. **粒度確認**: diff が 1 issue に収まっているか、跨いでいたら分割を提案 (これは停止条件)
6. 4 チェック適用 (結論先出し / 1 段落 1 主張 / 読み手語彙 / 儀礼削除)
7. PII scan
8. 出力

各ステップは「ユーザに確認 → 次へ」の対話形式で進める。一気に全文を出さない。

## 5 セクション固定 template

````markdown
## 課題

[背景・現状の問題、1-2 段落]

[この PR では、〜します の 1 文で結論先出し]

## DoD

- [完了基準を bullet で、各項目 testable]

## 実装の流れとレビュー順

1. [step 1 のタイトル]

   - File: `path/to/file.ext`
   - Change: [何を変えたか、過去形「〜しました」]
   - Review: [何を確認すべきか、「〜を確認してください」]

2. [step 2 のタイトル]

   - File: `...`
   - Change: ...
   - Review: ...

## 実装中に分かったこと

- [副次的に判明したこと]
- [scope 外と判断して別 issue / 別 PR に切り出したこと、PR 番号があれば参照]

## 検証

- `yarn exec tsc --noEmit --incremental false --pretty false`
- 変更ファイルに対する `yarn exec biome check`
- ブラウザでの手動会話テスト

手動確認:

- [具体的なテストシナリオ]
````

## 粒度ルール: 1 issue = 1 PR

- 実装中に発見した別問題は scope に含めず、「実装中に分かったこと」に記録 + 別 issue / 別 PR
- 関連性があっても、別 issue として独立して動く粒度なら分割
- 例: 「Knowledge ID 取得 tool の実装」中に「進捗文だけ最終回答 bug」を発見 → 別 PR に切り出し、両 PR で互いを参照
- diff が複数 issue に跨いでいたら **停止条件発動** (`sadoku` 本体が分割を提案)

## 4 チェック (最終工程で適用)

1. **結論先出し** — 1 文目で「何が変わったか」が分かる
2. **1 段落 1 主張** — 読み手に並列処理させない
3. **語彙が読み手のもの** — チーム / 外部レビュアーが知ってる言葉か
4. **儀礼削除** — 「まず最初に」「以上、よろしくお願いします」は技術文に不要

「自然な文章」を抽象的に求めない (判定は読み手任せで指針にならない)。AI 臭リストは作らない (model 世代で症状が変わる、固定リストは陳腐化、単語狩りは perverse incentive)。

## 図優先 (§3.8 引き算原則)

構造化できる情報は、散文より **表 / mermaid / box diagram を優先**。

- 「実装の流れとレビュー順」の step が 5+ なら、mermaid flowchart で全体像を冒頭に
- 「変更前後の対比」は表 (before/after の 2 列) で
- データ構造の変更は box diagram で
- 判断のニュアンス (なぜこの選択をしたか) は散文で残す

レビュアーが PR を初見で 30 秒以内に骨格を把握できる構成を目指す。散文だけで「読めば分かる」状態は引き算原則違反。

## PII / Secrets scan

PR 説明文 / commit message / release notes 出力前に最低限以下を grep:

```bash
# email
grep -E '[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}' <draft>
# token-like
grep -E '(sk-|ghp_|ghs_|xox[bao]p)-?[A-Za-z0-9_]{16,}' <draft>
# 個人名 (チーム外の実名)
# → 人手確認、機械では完全には判定できない
```

0 件でも完了記録に `PII scan: clean (0 matches)` と検証ログを記載する。
