1. 役割境界

動詞単位で 4 分割。skill 間の主な handoff は kouchiku → shiken (TDD 必要層) と kouchiku → sadoku (実装完了) の 2 本。tansaku はバグ発生時にどの skill からでも飛び込める。

見る・書く
sadoku — 査読
code review / PR 説明文 (実装は kouchiku に委ねる)
考える・作る
kouchiku — 構築
設計判断 / 評価 / 計画策定 / 計画実行 (考える→作るを一気通貫)
追う
tansaku — 探索
バグ調査 / root cause investigation
試す
shiken — 試験
TDD discipline / RED → GREEN → REFACTOR → PRUNE

2. 典型ワークフロー A: 新機能実装

issue 受領から PR 出荷まで。kouchiku が考える→作るを一気通貫、TDD 必要層では shiken に分岐、完了後に sadoku がレビュー。

issue + DoD
「設計どうする」
kouchiku 通常検討
問題定義 / 推奨案 + 代替案 / 前提崩し / 攻撃検証 / Plan steps 出力
Plan 承認 / 「進めて」
kouchiku 計画実行
step ごとに inline で実装 + 検証
TDD 必要層に触れたら一時停止
shiken
RED (fail を目視) → GREEN (pass を目視) → REFACTOR → PRUNE (revert で fail 目視)
サイクル完了 → 戻る
kouchiku 計画実行 (続き)
全 step 完了まで
↓
sadoku 通常レビュー
深さ判定 → 停止条件 → 専門家レビュー (Standard 以上) → 完了記録
Standard 以上で gate (b)
専門家レビュー (subagent 並列、最大 3)
reviewer-security / reviewer-architecture / adversarial (これだけ inline)
裏取り済の指摘を採用
sadoku PR 説明文
references/pr-template.md の 8 ステップを対話形式で進行
PII scan / 4 チェック
PR open
ポイント kouchiku → shiken → kouchiku の往復は同セッションで起こる。sadoku は実装完了後に初めて起動 (「見る」専門)。専門家レビューは Standard 以上のみ、Quick は停止条件チェックだけ。

3. 典型ワークフロー B: バグ修正

バグ報告から regression guard 付き PR まで。

バグ報告 / error / 「動かない」
↓
tansaku 通常追跡
1. 症状をそのまま列挙 / 2. hypothesis 1 文 / 3. instrument 1 つで confirm/discard (references/logging-techniques.md) / 4. confirm → fix / 5. fix 前後の挙動 diff を verbatim
root cause 確定、fix 実装の前に
shiken regression guard
RED: バグ再現 test (fail) / GREEN: fix 実装で pass / PRUNE: revert → fail 目視
↓
sadoku 通常レビュー + PR 説明文
↓
PR open
ポイント tansaku は「触らない」discipline (hypothesis を 1 文書けるまで code に触らない)。bugfix は shiken の強制層、必ず regression test を先行。「直りました」だけの自己申告は禁止、fix 前後の挙動 diff を verbatim 引用。

4. 典型ワークフロー C: 設計判断のみ

判断要求 → verdict、code は触らない。

「kill か keep か」「やる価値ある?」
↓
kouchiku 評価
Verdict: Kill / Keep / Pivot + 3 理由 (user 制約ベース) + (Pivot なら方向転換先)
ルール 「保留」は出さない (判断回避)。技術的な好みだけで決めない (user 制約を根拠に含める)。

5. 典型ワークフロー D: 小さい修正

scope < 3 file の即決パターン。

「どうやって直す」「やり方どっち」
↓
kouchiku 軽量検討
推奨 (file:line) + brute force + risk 1 つ
案を選択 / 「進めて」
kouchiku 計画実行
TDD 必要層なら shiken 経由
↓
sadoku 通常レビュー (Quick)
停止条件チェックのみ (50 行以内)
↓
PR open

8. NG パターン

NG-1: 廃止された「レビュー咀嚼モード」を skill で済ませる

reviewer コメント 5 件まとめて受領
↓
sadoku レビュー咀嚼 ?
v3.0 で廃止された mode を呼ぼうとする
✗ そんな mode は無い
通常会話で対応
「これらに返信書いて」「実装必要なものは kouchiku に渡して」

NG-2: TDD レイヤー判定をすっ飛ばす

純ロジック層に新規実装
↓
kouchiku 計画実行 (shiken スキップ)
強制層を素通りして直接実装
✗ sadoku の停止条件にひっかかる
sadoku 停止条件発動
テスト最小性違反 / 未テスト実装

NG-3: hypothesis なしで tansaku に入る

「とりあえず試そう」
↓
tansaku
Hard Rule 違反: hypothesis を 1 文書けるまで code を触らない
✗ instrument 仕込みすぎ / root cause 不明 / 再発
